# Performance Design: Caching и Rate Limiting

## Анализ производительности

### "Hot Paths" 

1. GET /api/hotels - Просмотр отелей

- Операция часто выполняется пользователями при поиске жилья. Скорее всего каждый зашедший пользователь выполнит эту операцию.
- Данные изменяются редко. Новые отели добавляются не очень интенсивно, поэтому не будет необходимости часто инвалидировать кеш.

Также сюда отнесится получение отелей по городу `GET /api/hotels?city=XXX` с теми же сооброжениями

2. GET /api/bookings - Просмотр бронирований пользователя

- Пользователи чаще проверяют свои бронирования, чем делают новые. Обновляются данные очень редко.
- Данные специфичны и зачастую статичны для пользователя.
- При перемещениям по своим бронированиям на сайте бронирования пользователя будут отображаться очень часто с одними и теми же данными.

### Медленные операции (обращения к БД)

- Поиск отелей по городу/отображение всех отелей - требует фильтрации в базе MongoDB, возвращает очень много данных.
- Создание бронирования пользователем - должна с блокировкой проверятся возможность бронирования на определенные даты. (В идеальном мире, у меня такого нету)

### Требования к производительности

- Время получения отелей 0.95q < 100ms. Поиск отелей должен работать быстро, чтобы удерживать пользователя на сайте.
- Время создания сущностей 0.95q < 300ms. Создание сущностей может работать медленее, так как эти сценарии происходят во много раз реже, чем просмотр.
- Пропускная способность 100+ RPS

## Кеширование

Для кеширования я использовал Redis.

В качестве стратегии кеширования я выбрал Cache-Aside. Он хорошо подходил мне, так как
у в моем сервисе есть множество запросов, которые возвращают один и тот же результат для нескольких пользователей.
Полный список таких операций:

#### GET /api/hotels - Список всех отелей

Идеально подходит для кеширования, так как все пользователи будут получать один и тот же набор данных. Чтобы каждый
раз не получать их из хранилища их можно закешировать.

- TTL 1 час
- Ключ в Redis `hotels:all`

#### GET /api/hotels?city=XXX - Отели по городу

Такая же ситуация как и с прошлым хендлером. Результат полученным одним пользователем можно переиспользовать для другого.

- TTL 1 час
- Ключ в Redis `hotels:city:{city}`

#### GET /api/bookings - Бронирования пользователя

Получение пользователем его бронированний будет тоже довольно частой операцией, гораздо больше чем действий с бронированием, поэтому можно добавить кеш.

- TTL 30 минут 
- Ключ в Redis `bookings:user:{userId}`

Стратегия инвалидации кеша:
- При добавлении нового отеля `POST /api/hotels` удалить данные из кеша, чтобы они собрались заново
- При создании нового бронирования `POST /api/bookings` удалить бронирования пользователя `bookings:user:{userId}`, чтобы они собрались заново
- При отмене бронирования `DELETE /api/bookings/{id}` удалить из кеша `bookings:user:{userId}`, чтобы броинрования собрались заново

## Rate Limiting

Я выбрал классический алгоритм token bucket, потому что у него одна из самых простых реализаций. Я поставил rate limit на следующие ручки, потому что они могут создавать большую нагрузку на систему пишущими операциями:

- POST /api/bookings (создание бронирования). Лимит 10 запросов в минуту на пользователя

- POST /api/hotels (создание отеля). Лимит 100 запросов в час на администратора


В случае, если запрос был отброшен, то возвращается код ответа 429.

В каждом ответе по ручке с RateLimit возвращаются заголовки
- `X-RateLimit-Limit: 10` (лимит запросов)
- `X-RateLimit-Remaining: 5` (оставшиеся запросы)
- `X-RateLimit-Reset: 1234567890` (Unix timestamp)

# Анализ производительности

### Кеширование

За счет кеширования большая часть запросов должна использовать уже готовые закешированные данные, поэтому Response Time ручек с кешем на 0.5q и 0.75q должен заметно уменьшится.

Также за счет снижения количества обращений к базе данных должна вырасти максимальная пропускная способность кешированных данных.

Для измерения эффективности нужно использовать:
- HitRate - % запросов, которые получили данные из кеша. Его можно собирать метриками. Считается как `cache_hit_rate (%) = hits / (hits + misses) * 100`. Я считаю метрики hits и misses внутри кеша. Затем их можно было бы собирать в prometheus.
- Response Time на небольших квантилях косвенно - если он остается маленьким, значит, данные успешно получаются из кеша.

### Rate Limiting

За счет кеширования должны отсекаться некоторые запросы. Соответственно основаня метрика тут будет количество отклоненных запросов и процент отклоненных запросов.

Большой процент отклоняемого трафика может говорить либо о неправильной настройке Rate limiter, либо о большом количестве злоумышленников (ботов).
